# 戰鬥模擬器 — 遊戲規則文件

> 本文件描述戰鬥規則本身，相當於遊戲設計文件（GDD）的戰鬥章節。
> 給企劃、玩家、規則討論者閱讀。
> 程式實作的 source of truth 在 `scripts/formulas.py`，文件僅作說明。

---

## 1. 設計目標

這套規則模擬「兩支部隊在地圖上某個地格相遇」的單場戰鬥。設計上想達成幾個目標：

- **可重現**：同樣的輸入 + 同樣的隨機種子 → 結果完全一致，便於平衡性測試
- **有運氣成份但可控**：透過「穩定度」屬性讓玩家在「穩定但平庸」與「波動但爆發」之間做策略選擇
- **戰力差異有意義**：強隊不應 100% 免疫弱隊反擊，弱隊也不能毫無還手之力
- **戰報可讀**：每個數值的由來都能在戰報中追溯到公式

## 2. 核心概念

### 2.1 手下（Minion）

戰鬥的最小單位。每個手下有三個屬性：

| 屬性 | 範圍 | 意義 |
|---|---|---|
| `power`（戰力值） | 1～9999 | 同時是攻擊力與生命值。受傷會扣減，治療會回復，但永遠不超過原始值 |
| `stability`（穩定度） | 0.0～0.99 | 戰力抽樣的波動範圍。0 = 絕對穩定；0.15 = ±15%；0.99 = ±99% |
| `skills`（技能列表） | 0～4 個 | 在 BC 或 AC 階段觸發的特殊效果，可為空 |

**穩定度的直覺**：穩定度越低，這個手下越「可預測」；穩定度越高，越「靠運氣」。一個 σ=0 的手下無論抽樣多少次，戰力都是固定值；σ=0.5 的手下每次抽樣可能在 50%～150% 之間飄。

### 2.2 隊伍（Team）

由 1～3 個手下組成。隊伍沒有獨立屬性，所有計算都從手下層級加總而來：

- **隊伍總戰力 = Σ 各手下當下戰力**（注意：是「當下」不是「原始」，受傷後會減少）
- **隊伍抽樣戰力 = Σ 各手下個別抽樣值**（每個手下用自己的 σ 抽樣後加總）

### 2.3 技能（Skill）

技能屬於某個手下，定義在 `data/skill_library.yaml` 圖鑑中，由手下的 `skills` 欄位引用 ID。

| 屬性 | 意義 |
|---|---|
| `phase` | `before_combat` 或 `after_combat`，決定在哪個階段觸發 |
| `effect_type` | `damage_enemy`（對敵傷害）或 `heal_self`（對己治療） |
| `value` | 效果數值（傷害量或治療量） |
| `activation_rate` | 發動率，0.0～1.0。未填預設 1.0（必發動） |

**目前不支援**：buff/debuff、條件觸發（如 HP < 50%）、多回合冷卻、AoE/單體區分。所有技能效果都對「整個敵隊」或「整個己隊」生效。

### 2.4 戰鬥階段（Phase）

一場戰鬥由三個階段依序進行：**Before Combat → Combat → After Combat**。

不重複、不迭代——一場戰鬥就是這三個階段各跑一次。

## 3. 戰鬥流程

### 3.1 Before Combat（戰前階段）

雙方根據 BC 段技能，**同時**結算對敵傷害與對己治療。

「同時結算」意味著沒有先後手——A 與 B 的效果用「進入此階段時的戰力」作為基準同時計算，最後一次性套用到雙方戰力上。

#### 範例

A 隊 BC 技能對敵造成 10 傷、B 隊 BC 技能對敵造成 5 傷，雙方初始戰力 50/40：

```
進入 BC 時：A=50, B=40
A 對 B 造成 10 傷（用 A=50 為基準計算）
B 對 A 造成 5 傷（用 B=40 為基準計算）
同時套用 → A=45, B=30
```

#### 技能發動判定

每個技能在本場戰鬥中最多觸發一次。若技能定義了 `activation_rate`，則：

```
擲一個 [0, 1) 的隨機數 roll
若 roll < activation_rate → 發動
否則不發動
```

未發動的技能本場不再判定。

#### 傷害分攤到手下

當隊伍承受 X 點總傷害時，按進入該階段的當下戰力比例分配給各手下。例如 A1=60、A2=40，A 隊承受 30 傷，則 A1 扣 18、A2 扣 12。

戰力為 0 的手下（已倒下）不參與分攤，也不會被治療復活。

### 3.2 Combat（戰鬥階段）

核心傷害交換階段，使用 BC 修正後的戰力作為輸入。

#### Step 1：穩定度抽樣

每個還活著的手下根據自己的 σ 對戰力抽樣：

```
sampled_power_i = power_i · U(1 − σ_i, 1 + σ_i)
```

其中 `U(a, b)` 是均勻分布。隊伍抽樣總和：

```
sampled_team = Σ sampled_power_i
```

#### Step 2：計算減傷比例

```
mitigation_X = clamp( sampled_X / (raw_power_A + raw_power_B), 0.1, 0.9 )
```

注意分母用「雙方原始（未抽樣）戰力總和」。雙方 mitigation **各自獨立**夾在 [0.1, 0.9]。

**減傷夾值的意義**：
- 上限 0.9 → 強勢方天花板，確保攻方至少打出 10% 戰力的傷害
- 下限 0.1 → 弱勢方保底，確保至少擋掉 10% 傷害

夾值範圍可在 `data/global_config.yaml` 調整。

#### Step 3：計算傷害

```
damage_A→B = raw_power_A · (1 − mitigation_B)
damage_B→A = raw_power_B · (1 − mitigation_A)
```

注意這裡用攻方的「原始（當下未抽樣）戰力」，抽樣的影響只反映在 mitigation 上。

#### Step 4：套用傷害

按比例分攤到各手下，戰力夾在 `[0, 原始戰力]`。

#### 完整範例

```
A: A1(power=80, σ=0.10), A2(power=50, σ=0.20)  → 原始總戰力 130
B: B1(power=90, σ=0.15), B2(power=40, σ=0.10)  → 原始總戰力 130

Step 1: 抽樣（假設 RNG 抽出以下倍率）
  A1: 80 · 1.05 = 84
  A2: 50 · 0.92 = 46
  B1: 90 · 0.88 = 79.2
  B2: 40 · 1.06 = 42.4
  sampled_A = 130, sampled_B = 121.6

Step 2: 減傷（分母 130+130=260）
  mitigation_A = clamp(130/260, 0.1, 0.9) = 0.500
  mitigation_B = clamp(121.6/260, 0.1, 0.9) = 0.468

Step 3: 傷害
  damage_A→B = 130 · (1 − 0.468) = 69.2
  damage_B→A = 130 · (1 − 0.500) = 65.0

Step 4: 分攤
  A1 承受 65 · (80/130) = 40.0 → A1 power: 80 → 40
  A2 承受 65 · (50/130) = 25.0 → A2 power: 50 → 25
  B1 承受 69.2 · (90/130) = 47.9 → B1 power: 90 → 42.1
  B2 承受 69.2 · (40/130) = 21.3 → B2 power: 40 → 18.7
```

### 3.3 After Combat（戰後階段）

規則同 BC 段，使用「進入 AC 時的當下戰力」（即 Combat 結束後的狀態）作為計算基準。

雙方同時結算技能效果，同樣按戰力比例分攤。治療上限為手下原始戰力，溢出捨棄。

## 4. 公式特性與設計選擇

### 4.1 為什麼分母用「原始戰力」而非「抽樣戰力」？

兩種設計的差異：

- 用原始：`mitigation_A + mitigation_B` 會浮動，雙方都抽到高值時整體減傷高（兩敗俱輕傷），都抽低時整體減傷低（血流成河）
- 用抽樣：`mitigation_A + mitigation_B ≡ 1`，運氣只決定誰佔上風，整體傷害量恆定

我們選擇**用原始戰力**，因為它讓「穩定度」這個屬性真正有意義——穩定度同時影響「這場戰鬥的整體烈度」與「誰佔上風」。配合 [0.1, 0.9] 的夾值防止極端情況。

### 4.2 為什麼傷害用「原始戰力」而非「抽樣戰力」？

抽樣值的影響全部反映在 mitigation 上。如果傷害也用抽樣，那會雙倍放大運氣的影響——抽到神運氣的手下會傷害飆高 *且* 對手減傷變低。我們希望運氣是「單一維度」的影響，所以傷害用原始戰力。

### 4.3 BC/AC 為什麼同時結算？

避免「先手優勢」這個隱藏變數。規則設計上每個手下、每個隊伍都應該被公平對待，「先發動的人佔便宜」會讓策略簡化成「拚誰的技能先觸發」，失去多元性。

## 5. 隨機性與 Seed

### 5.1 全域共用 seed

所有 RNG 抽樣（穩定度抽樣、技能發動率擲骰）共用一個 seed 序列。**同樣的 seed + 同樣的輸入 → 結果完全一致。**

### 5.2 RNG 抽樣順序

順序固定，確保 determinism：

1. BC 段：A 隊技能擲骰（A1 → A2 → A3 各自的技能 by 順序）
2. BC 段：B 隊技能擲骰
3. Combat 段：A 隊穩定度抽樣（A1 → A2 → A3）
4. Combat 段：B 隊穩定度抽樣
5. AC 段：A 隊技能擲骰
6. AC 段：B 隊技能擲骰

調換手下順序會改變抽樣序列，可能導致不同結果。

### 5.3 Seed 從哪裡來

**單場戰鬥**：seed 寫在 `data/global_config.yaml`，可用 `--seed` 參數覆寫。

**批次模擬**：依 `data/analysis_config.yaml` 的 seed_strategy 產生：

| 模式 | 產生方式 | 重現性 |
|---|---|---|
| `sequential`（預設） | 等差數列 base, base+step, ... | 整批可重現 |
| `derived` | master_seed 衍生 | 整批可重現 |
| `random` | 真隨機 | 不可重現 |

## 6. 名詞速查

| 術語 | 英文 | 說明 |
|---|---|---|
| 戰力值 | Power | 同時擔任攻擊力與生命值 |
| 原始戰力 | Original / Raw Power | 手下進入戰鬥時的戰力，是治療上限 |
| 當下戰力 | Current Power | 隨戰鬥變動的即時值 |
| 抽樣戰力 | Sampled Power | 由穩定度產生的隨機波動值，用於減傷計算 |
| 穩定度 | Stability / σ | 抽樣波動範圍的百分比 |
| 減傷比例 | Mitigation | Combat 段抵擋多少比例傷害 |
| 發動率 | Activation Rate | 技能觸發機率 |
| BC | Before Combat | 戰前階段 |
| AC | After Combat | 戰後階段 |

## 7. 已知限制與未來擴展空間

當前版本是 MVP，刻意保持簡單。下列是預期的擴展方向：

- **連續戰鬥**：當前單場戰鬥結束就終止，戰力不延續到下一場。未來支援多場連戰，需要記錄 `present_power / max_power` 兩個值
- **技能類型擴展**：當前只有 damage_enemy / heal_self。未來可加入 buff（攻擊強化）、debuff（穩定度懲罰）、condition（HP < X% 才觸發）
- **戰鬥環境（地形）**：當前無地形效果。未來可在戰鬥輸入中加入 terrain 修正
- **技能目標**：當前所有技能都對全隊生效。未來可加入單體目標選擇
- **連鎖反應**：當前 BC/AC 同時結算。若引入「反擊」「連動」等機制，需要重新設計階段
